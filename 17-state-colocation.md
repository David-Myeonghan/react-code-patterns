# 17. State Colocation

상태 배치 최적화 — 상태를 사용하는 곳에 최대한 가까이 배치하여 불필요한 리렌더링 범위를 최소화하는 패턴

## 동작 원리

React에서 상태가 업데이트되면 해당 상태를 보유한 컴포넌트와 **그 아래 모든 자식 컴포넌트**가 리렌더링된다. State Colocation은 이 원리를 역으로 활용하여, 상태를 가능한 한 **실제로 사용하는 컴포넌트에 가깝게** 배치함으로써 리렌더링 범위를 최소화한다.

상태 배치 우선순위: 전역 스토어 → Context → 부모 컴포넌트 → **해당 컴포넌트** 순으로 가능한 한 아래로 이동시킨다. 상태가 상위에 있을수록 리렌더링 범위가 넓어지므로, 상태를 사용하지 않는 컴포넌트까지 불필요하게 리렌더링되는 것을 방지할 수 있다.

## 장점

- **리렌더링 범위 최소화**: 상태 변경 시 관련 없는 컴포넌트의 리렌더링 방지
- **추가 라이브러리 불필요**: `React.memo`, `useMemo` 없이도 성능 개선 가능
- **코드 가독성 향상**: 상태와 그 상태를 사용하는 로직이 같은 위치에 있어 이해하기 쉬움
- **유지보수 용이**: 컴포넌트를 삭제/이동할 때 관련 상태도 함께 이동

## 단점/주의사항

- **과도한 분리 주의**: 모든 상태를 최하위로 내리면 나중에 형제 컴포넌트 간 상태 공유가 필요할 때 "상태 끌어올리기(lifting state up)"가 필요
- **공유 상태와의 균형**: 여러 컴포넌트가 동일한 상태를 필요로 하면 적절한 공통 조상에 배치해야 함
- **리팩토링 비용**: 요구사항 변경으로 상태 공유 범위가 달라지면 상태 위치를 옮겨야 함

## 적합한 사용 사례

- 검색 입력 필드의 쿼리 상태 (검색 컴포넌트 내부에 배치)
- 모달/드롭다운의 열림/닫힘 상태 (해당 컴포넌트 내부에 배치)
- 폼 필드의 유효성 검사 상태 (각 필드 컴포넌트 내부에 배치)
- 토글, 아코디언 등 독립적인 UI 상태

## 구현 예제

```tsx
// BAD — 상위에 불필요한 상태
function App() {
  const [query, setQuery] = useState('')  // App 전체가 리렌더링됨
  return (
    <Header />
    <SearchBar query={query} setQuery={setQuery} />
    <ExpensiveList />  {/* query와 무관한데 리렌더링됨 */}
  )
}

// GOOD — 상태를 사용하는 곳으로 이동
function App() {
  return (
    <Header />
    <SearchBar />        {/* 내부에서 자체 state 관리 */}
    <ExpensiveList />    {/* 리렌더링 안 됨 */}
  )
}
```
